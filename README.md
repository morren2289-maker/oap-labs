# Лабораторна робота №5

## Дослідження та усунення вразливостей веб-застосунку

### Мета роботи

Ознайомитися з типовими вразливостями веб-застосунків, відтворити їх на практиці та реалізувати заходи захисту.

---

# Сценарій А. SQL Injection

## Вразлива версія

У початковій реалізації SQL-запити формувалися шляхом конкатенації рядків.

Приклад:

```ts
return await get(`
  SELECT *
  FROM Users
  WHERE email='${email}';
`);
```

Користувач міг передати спеціально сформований рядок:

```sql
' OR 1=1 --
```

У результаті запит перетворювався на:

```sql
SELECT *
FROM Users
WHERE email='' OR 1=1 --';
```

Умова `1=1` завжди істинна, тому запит повертав усі записи.

### Наслідки

* обхід логіки пошуку;
* витік даних;
* можливість модифікації запитів.

---

## Виправлення

Було використано параметризовані запити SQLite.

Після виправлення:

```ts
return await get(
  `
  SELECT *
  FROM Users
  WHERE email = ?;
  `,
  [email]
);
```

Значення передається окремо від SQL-коду та не може бути виконане як частина запиту.

---

## Перевірка після виправлення

Повторний запит:

```sql
' OR 1=1 --
```

більше не змінює структуру SQL-запиту та обробляється як звичайний текст.

SQL Injection не відтворюється.

---

# Сценарій B. IDOR (Insecure Direct Object Reference)

## Вразлива версія

У початковій реалізації будь-який користувач міг отримати відгук за його ідентифікатором:

```ts
const review =
  await reviewsRepository.getById(id);
```

Перевірка власника запису відсутня.

Наприклад:

```
GET /api/v1/reviews/1
```

повертав відгук незалежно від того, хто виконує запит.

---

## Наслідки

* перегляд чужих даних;
* доступ до ресурсів інших користувачів;
* порушення конфіденційності.

---

## Виправлення

Було додано визначення поточного користувача через заголовок:

```
X-Demo-UserId
```

та перевірку власника ресурсу:

```ts
export async function getByIdForUser(
  reviewId: number,
  userId: number
) {
  return await get(
    `
    SELECT *
    FROM Reviews
    WHERE id = ?
      AND userId = ?;
    `,
    [reviewId, userId]
  );
}
```

Контролер використовує поточного користувача:

```ts
const currentUserId =
  (req as any).currentUserId;

await reviewsService.getReviewByIdForUser(
  Number(req.params.id),
  currentUserId
);
```

---

## Перевірка після виправлення

### Без заголовка

Запит:

```
GET /api/v1/reviews/1
```

Результат:

```json
{
  "error": "Unauthorized"
}
```

HTTP статус:

```
401 Unauthorized
```

### Чужий користувач

Запит:

```
GET /api/v1/reviews/1
X-Demo-UserId: 2
```

Результат:

```json
{
  "code": "FORBIDDEN",
  "message": "Access denied"
}
```

HTTP статус:

```
403 Forbidden
```

IDOR більше не відтворюється.

# Додаткові заходи безпеки

## Єдиний формат помилок

Реалізовано централізовану обробку помилок:

```ts
return res.status(status).json({
  code: err.code || "INTERNAL_SERVER_ERROR",
  message: err.message || "Internal server error",
  details: err.details
});
```

Усі помилки повертаються в однаковому форматі.

---

## Налаштування CORS

Доступ дозволено лише з дозволених джерел:

```ts
app.use(cors({
  origin: [
    "http://localhost:5500",
    "http://127.0.0.1:5500"
  ]
}));
```

Використання глобального доступу через `*` не допускається.

---

# Висновок

У ході роботи було досліджено вразливості SQL Injection та IDOR. Для усунення SQL Injection використано параметризовані запити. Для захисту від IDOR реалізовано перевірку власника ресурсу на сервері та механізм визначення поточного користувача через заголовок X-Demo-UserId. Також налаштовано централізовану обробку помилок і безпечні параметри CORS.
